# 임시 아이디어

아직 정리되지 않은 러프한 메모. 나중에 [[RAG 시스템 설계]] 문서로 승격시킬 후보다. #inbox #idea

## RAG 파이프라인 개선 아이디어

- 질문이 들어올 때 검색 결과가 빈약하면, 곧바로 답을 만들지 말고 **재질의(query rewriting)** 를 한 번 더 돌려보자. #rag
- 검색 단계에서 키워드와 벡터를 같이 쓰는 방식이 확실히 낫다. 자세한 건 [[Hybrid Search]] 참고.
- 임베딩 비용이 걱정된다면 Mock provider로 먼저 흐름만 검증하고, 나중에 진짜 모델로 교체하는 편이 안전하다.

### 확인이 필요한 것

- 소비자(consumer)가 실패했을 때 몇 번까지 재시도하는 게 적절한가?
- 최종 실패 이벤트는 어디에 쌓아둘 것인가? → 아마도 죽은 편지함 패턴이 답일 듯.
