# Dead Letter Queue

죽은 편지함(Dead Letter Queue, DLQ)은 정상 처리에 반복 실패한 메시지를 따로 모아두는 별도의 큐 또는 토픽이다. 실패한 이벤트가 파이프라인을 막지 않도록 격리하는 것이 목적이다. #dlq #reliability

## 왜 필요한가

메시지 처리는 여러 이유로 실패한다. 스키마가 깨졌거나, 외부 API가 응답하지 않거나, 데이터 자체가 잘못됐을 수 있다. 이런 이벤트를 무한 재시도하면 소비자가 한 메시지에 묶여 뒤따르는 정상 이벤트까지 처리하지 못한다. DLQ는 "더는 자동으로 처리할 수 없는" 이벤트를 안전하게 빼내어 흐름을 지킨다. #dlq

## 이동 규칙

이 프로젝트에서는 [[Kafka]] 소비자가 다음 순서를 따른다.

1. **로컬 재시도**: 즉시 몇 차례 재시도한다.
2. **재시도 토픽**: 지연 후 다시 처리한다.
3. **재시도 횟수 초과**: 정해진 최대 재시도 횟수를 넘기면 `rag.dead-letter` 토픽으로 옮긴다.

### 메타데이터 보존

DLQ로 보낼 때는 원본 메시지에 더해 실패 원인, 마지막 예외 메시지, 누적 재시도 횟수를 함께 기록한다. 이 정보가 있어야 나중에 원인을 진단할 수 있다.

## 재처리

DLQ는 종착지가 아니라 대기실이다. 운영자가 원인을 고친 뒤(예: 스키마 수정) 이벤트를 원래 토픽으로 다시 흘려보내면 정상 처리된다. 전체 장애 처리 맥락은 [[RAG 시스템 설계]] 의 "장애 처리" 절을 참고한다. #reliability
